The benchmark is honest — the table is misleading

A forensic pass over BenchmarkZkcExecKeccakV2*: every surprising delta has a measured explanation, no caching artefacts inflate gogen, and two cheap changes (binary I/O, PGO) cut gogen wall time roughly in half on both shapes. Rev 2 adds what the benchmarks now report: executed micro-instructions and Minstr/s (§08), a cross-workload sweep — sort, fnv1a, blake2f (§09) — and the interpolation to the production target, a RISC-V interpreter executing an EVM block (§10).

Bottom line. The numbers reproduce on this machine (bytecode 14.2 s, gogen 208 ms plain; gogen 175 ms lowered) and the deltas that looked suspicious are exactly what the per-block arithmetic predicts. Puzzle from the PR tableResolution (measured)
WordMachine: 24 s plain → 85 s lowered, “only 3.5×”? Input shrank 100×. Per block it is 485 µs → 170.8 ms = 352× ≈ 172× more instructions × ~2× for Uint (big-int) words vs uint64.
gogen: 218 ms plain → 189 ms lowered, “lowered is faster”? Same 100× input gap. Per block 4.4 µs → 379 µs = 87× slower. On pure execution (harness stripped) it is 2.4 µs → 330 µs = 139×, close to the 172× step blowup — the residual gap is Go inlining digesting the helper-call storm better than an interpreter.
Why is the lowered baseline so much weaker? Only the Uint (heap big-int) WordMachine can interpret the lowered shape (u65 temporaries); the u64-narrowed tiers skip. So the lowered “oracle” is the slowest possible interpreter — quoting “451×” against it overstates gogen vs a like-for-like u64 tier.
Tip: the benchmark already emits a comparable metric — b.SetBytes gives MB/s (gogen: 32.6 plain vs 0.39 lowered = the honest 84×). Quote MB/s or µs/block in PR tables, never raw ns/op across these two benchmarks.

The interpreter tiers receive pre-decoded, in-process inputs — the decode and spawn columns are a tax only gogen pays, inside the timed loop. On the lowered benchmark the input is small, so >95% of its 189 ms is genuine execution.

Codegen is tight — the ~5× gap to hand-written keccak is the program's memory round-trips per lane op (state lives in ZkC RAM, 271 access sites), not emitter slack.

Lowered profile — death by recursion

fn__bit_shr_u64       30%  ← 1 bit per CALL (!)

fn__bit_xor_u2/4/8/32 22%  ┘ u64 → 2×u32 → … → 64×u1

A 63-bit rotate = 63 recursive call frames; one u64 XOR ≈ 63 calls × ~16 instructions each. Inherited from LowerNatives, faithfully compiled.

06Plan — ordered by leverage

1 · Benchmark hygiene ✓ done on this branchno semantics
Every sub-benchmark now reports instrs/op + Minstr/s (the size-independent, cross-tier metric — see §08); build_ms fixed (memoized first real build); gogen cells warm the freshly built binary before the timer; BenchmarkZkcExecSweep adds three contrasting workloads (§09). Still recommended for headline PR tables: -count 5 + benchstat.
2 · Binary I/O protocol for the generated harness 1.8× plain e2eeffort Sdeferred — prod
Length-prefixed name + raw LE u64 frames on stdin/stdout. Measured: decode 70 → 1.3 ms. Deferred for now: JSON is the input contract users and tests rely on for CLI/differential work — keep JSON as the default and add the binary path when the prod/tracer harness lands (it is also the natural wire format for checkpoint snapshots to shard workers).
3 · PGO in the build cache 2.2× lowered e2eeffort S–Mdeferred — prod
Teach gogen.Cache an optional two-step build: compile, run once on a small representative input with CPUPROFILE, rebuild with -pgo. Content-addressed like the binary itself. Pure Go-compiler optimisation — generated source untouched. Not integrated now; the measured 2.05× on the prover shape is banked for the prod build path.
4 · Codegen inlining of leaf helper calls ≥2× lowered, deterministiceffort M1:1 preserved
The $bit_xor_u1-style leaves are tiny pure functions; gogen can splice their bodies at call sites (registers renamed) instead of relying on PGO to discover it. Same transformation Go's inliner already performs legally — but deterministic, profile-free, and stackable with PGO. CALL remains a CALL in the instruction stream; only the emitted Go flattens, so the macro-PC ↔ statement mapping ([vi.ci] comments) survives.
5 · In-process AOT execution (already planned) −6–8 ms/invocationeffort M–Lno semantics
Spawn + pipe is 9% on one 50k-block run but becomes dominant when a sharded tracer dispatches many small shard re-executions. go build -buildmode=plugin or a persistent worker pool speaking the binary protocol both remove the floor; the worker pool also amortises one process per program, not per shard.
6 · Two-tier sharded tracing (design work) pays the 172× only where traceddesign1:1 critical
The lowered shape costs 172× more instructions by construction — that is the trace the prover needs, so it cannot be optimised away where columns are materialised. The architecture that wins: run the fast shape ahead to drop checkpoints (memory snapshot + machine state at macro-PC boundaries), then re-execute only the shard being traced with the faithful lowered program, in parallel across shards. Items 2–5 make both tiers faster without touching this contract. Anti-goal: folding $bit_* semantics into native ops inside a traced shard — that deletes prover-visible register writes.
7 · Parked: scalar promotion of small RAMs ~5× plain ceilingeffort Lrisky
The residual gap to hand-written keccak is state living in ZkC RAM (memGet/memWrite per lane op). Promoting small fixed-extent memories to locals would close it — but memory contents are machine state that checkpoints must observe, so it needs flush-at-boundary semantics. Revisit after the checkpoint design exists; don't pre-empt it.

What the sweep establishes

ObservationEvidence
gogen's rate is workload-stable at ~10–11 Ginstr/s on any dense workload — a single number you can interpolate with (~0.1 ns per ZkC micro-instruction). keccak 11.3, fnv1a 11.0, blake 10.3 Ginstr/s on 180M–2B-instruction lowered runs; plain keccak pure-execution rate 9.8.
Interpreter rates are also stable — and 2 orders of magnitude lower, regardless of workload. wordmachine-Uint: 24–48 Minstr/s; bytecode: 57–81 Minstr/s across all programs and shapes.
The lowering blowup is the workload-dependent variable: ×1.6 to ×306. sort ×1.6 (no native bitwise ops), keccak ×172, blake ×232, fnv1a ×306 (u128 ops).
Bytecode-tier coverage is the exception, not the rule: any u128 register (or u65 from lowering) disables it. Of 8 sweep cells, bytecode runs 2 (sort plain + sort lowered). gogen runs all 8.
gogen vs the only other tier that runs the prover shape: 409–459× on bit-heavy workloads, 86× on sort. fnv1a lowered 11.49 s → 25 ms; blake 7.20 s → 17.6 ms; sort 824 ms → 9.6 ms.

10Interpolating to production — the EVM-block scenario

The target workload: a RISC-V interpreter (written in ZkC) executing an EVM block — roughly 30× the keccak-50k benchmark, i.e. ~35 B plain-shape micro-instructions. With workload-stable rates, time is just instrs × blowup / rate:

Plain shape — 35 B instrs

TierRateEst. time
gogen~10 Ginstr/s~3.5 s
bytecode77 Minstr/s~7.5 min
wordmachine-Uint~35 Minstr/s~17 min

Prover shape — 35 B × blowup M

MlikegogenUint interp.
×2control/memory-bound~7 s~49 min
×30mixed ALU + memory~1.7 min~12 h
×170bit-op-bound (keccak-like)~10 min~2.9 days
Read it this way: on the prover shape, gogen is the only tier in viable territory — and the spread of M (×2…×170 depending on instruction mix) moves the answer more than any executor optimisation will. Two consequences for the roadmap: (1) measure M for the actual RISC-V interpreter early — it bounds everything; (2) the items that shrink M's cost (PGO ~2×, codegen inlining of $bit_* leaves, and ultimately sharding the faithful re-execution across checkpoints) are worth more than micro-tuning the plain shape. A worked check of the formula: keccak-50k plain = 1.167 B / 9.8 G = 119 ms — exactly the measured run time.

11x86 production preview — c8a.8xlarge (Zen 5)

partial data — run stopped early

A first pass on the production target (AWS c8a.8xlarge, AMD Zen 5) got through the keccak plain cells before being stopped (the wordmachine-lowered cell alone is minutes per iteration). The pattern is already consistent — single-core x86 is 25–35% slower than the M5 Max dev machine, across every tier:

keccak · plain · 50k blocksM5 Maxc8a (Zen 5)x86 / Mac
wordmachine-u6424.2 s · 48 Minstr/s36.2 s · 32.3 Minstr/s0.67×
bytecode15.1 s · 77.5 Minstr/s20.1 s · 58.0 Minstr/s0.75×

Notably the gap shrinks as the tier gets faster (0.67× → 0.80×): interpreter dispatch leans on single-thread latency where Apple's core is strongest, while gogen's straight-line generated code ports better. Extrapolating the unmeasured cells with the per-tier ratios (gogen ×0.8, interpreters ×0.7): gogen lowered ≈ 8.5–9 Ginstr/s, wordmachine-Uint lowered ≈ 16 Minstr/s on this box.

⚠ Production forecast (preliminary). Re-running the §10 EVM-block scenario (35 B plain-shape micro-instructions) on c8a single-core: gogen ~4.5–8 s plain (pure-execution vs end-to-end harness), bytecode ~10 min, Uint interpreter ~30+ min. On the prover shape at blowup M=30: gogen ~2 min, Uint interpreter ~18 h; at keccak-like M=170: gogen ~12 min, interpreter ~4+ days. Three caveats before treating these as the plan of record: (1) data is partial — lowered cells and the sweep were extrapolated, not measured; (2) the x86-only levers are still unmeasured — PGO (2.05× on Mac lowered), GOAMD64=v3/v4 (Zen 5 has AVX-512) and bounds-check flags could claw back much of the 0.8× gap; (3) every executor here is single-threaded — the c8a.8xlarge's other 31 vCPUs are idle. The forecast that actually matters is per-shard: the checkpoint/sharded-tracer design (§06, item 6) is what turns “gogen ~12 min single-core” into “~25 s across 32 shards”, and these per-core rates are its sizing input.
